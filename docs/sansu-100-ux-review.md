# 100マス計算 UXレビュー & 変更計画

実機相当（iPhone 幅）で本番（https://snap-content.takumi-oda.com/sansu-100 ）を
新規テストユーザーで一通り操作した結果のレビュー。対象フロー:

登録（なまえ→アバター→いろ→あいことば）→ ホーム → レベル選択 → プレイ（正答/誤答/スキップ）
→ リタイヤ（部分保存）→ 履歴（グラフ・一覧・バッジ）→ 結果画面（デバッグ完走で確認）。

実施日: 2026-06-26 / 検証者: Claude（Computer Use + 手動操作）

---

## サマリ

全体としてよく作り込まれている（色分けレベル、○式の正答カウント、結果画面の祝福演出、
バッジ49種、部分保存、PIN確定ボタンなど）。一方で **子供が「間違えたとき」に何も起きない**
のが最大の課題。加えて、デイリー削除に伴う後始末漏れ・リタイヤ記録の重複など、
直近の変更に紐づく不整合が残っている。

優先度の高い順: **B2（誤答無反応）> B1（リタイヤ重複）> U2/U3（デイリー残骸・記録水増し）> その他**。

## 不具合（Bugs）

### B1【中】リタイヤ確定の連打で重複セッションが生成される
- 再現: プレイ中に「やめる」→確認ダイアログの「やめる」を素早く2回タップ。
- 結果: 別IDの部分セッションが2件保存される（localStorage `sansu-100:sessions:<id>` に
  `isRetired:true` が2件）。子供は連打しがちなので現実的に起きる。
- 原因: `app/sansu-100/play/page.tsx` のリタイヤ `onClick` が `if (finalized || retiring) return;`
  で守られているが、`setRetiring(true)` は非同期 state 更新のため、同期的な2連打では
  2回目も `retiring===false` のまま通過し、`finishSession()` が新IDで2回走る。
- 補足: 完走側（`useEffect` の `finalized` ガード）は比較的安全だが、リタイヤ側の onClick は脆弱。

### B2【中】誤答時に無反応で、間違いに気づけない（最重要UX）
- 再現: 「6 + 3 = __」に「5」など誤った数字を1つ押す。
- 結果: 解答欄に数字は入るが **判定されず・フィードバックも出ず・先にも進まない**。
  子供は手が止まる。間違いを直す（←）かスキップするしか進めず、しかも「間違えた」という
  学習フィードバックも得られない。
- 原因: `NumberKeypad`/`play` の自動判定が「入力が正解と一致したとき」だけ発火する設計
  （`handleInputChange` で `n === answer` のときのみ `judge`）。答えが1桁でも「2桁目を待つ」
  ため、誤った1桁では確定できない。
- 影響: 子供向けアプリの根幹（間違えても次へ・間違いを知る）が機能していない。

### B3【低】本番で `?debug=1` を付けるとデバッグ完走ボタンが出る（整合性）
- 再現: `/sansu-100/play?debug=1` でレベル選択 → 画面下に「一気に100問パーフェクト」等の
  DEBUG パネルが表示され、ワンタップで満点記録を生成できる。
- 原因: `debugMode = params.get('debug')==='1' || process.env.NODE_ENV !== 'production'`。
  本番でもクエリで有効化できてしまう。
- 影響: 偽のベスト記録・バッジ量産が可能。ランキング等を入れるなら要対策。

## UX・設計の改善点

### U1【中】テーマ色を選ぶがUIにほぼ反映されない
- 登録3ステップ目で「すきな いろ」を選ぶが、ホーム/プレイ/結果いずれの配色にも反映されない
  （ボタンは常に青）。選ぶ意味が薄く、登録の手間だけ増えている。
- 対応案: アバター枠・ヘッダー・スタートボタン等のアクセントに `themeColor` を反映する。
  反映しないなら登録ステップから外す。

### U2【中】削除した「デイリー」機能のバッジが取得不能のまま残存
- 「きょうのチャレンジ」を削除済みだが、バッジカタログに `daily_first/daily_7/...` 等の
  デイリー系バッジが残り、コレクションに**永久に解放されない ??** が並ぶ。
- 対応案: デイリー系バッジをカタログから除外、または別の達成条件に再割り当て。
- 関連ファイル: `app/sansu-100/lib/badge-catalog.ts` / `badges.ts`。

### U3【中】リタイヤ（数問）でもセッション1回＋バッジ獲得扱い
- 3問でリタイヤしても `totalSessions` が1増え、「はじめの一歩」等のバッジまで付与される。
  記録・バッジが安易に水増しでき、達成の価値が薄れる。
- 対応案: リタイヤ（`isRetired`）は「解いた問題の記録」としては残しつつ、
  `totalSessions` のカウントやバッジ判定の母数からは除外（または最低問題数の閾値を設ける）。

### U4【低】レベル番号が演算×難易度で交互配置され、進める順が分かりにくい
- 例: Lv.5 かけ算 → Lv.6 わり算 → Lv.7 たし算Lv.3 …と難易度と演算が交互。
- 対応案: 演算ごとにグルーピング表示、または「初級→上級」の明確な並びにする。色分けは良い。

### U5【低】レベル選択にベストタイム/クリア状況が出ない
- 各レベルカードに自己ベストや「クリア済み★」が無く、再挑戦の動機・達成感が弱い。
- 対応案: `bestTimesByLevel` をカードに小さく表示（例: ⏱ 0:58 / ★）。

### U6【低】履歴グラフがセッション1件だと軸が不自然
- 1件だけだと折れ線グラフの軸が 1→2 になり、ほぼ空のグラフが表示される。
- 対応案: 2件未満は「まだ グラフは でないよ。もっと れんしゅうしてね」等の空状態に。

### U7【低】登録の確定ボタンの処理中ラベルが文脈とズレる
- 登録処理中も `PinPad` 既定の「たしかめてるよ...」が出る（PIN照合用の流用）。
- 対応案: 登録時は「とうろくしてるよ...」を出す（`PinPad` に処理中ラベルを渡せるように）。

### U8【低】4桁あいことばの再入力確認が無い
- 打ち間違えても気づけない（復旧は管理CLIのみ）。子供＋保護者運用なら確認1回入れる価値あり。
- 対応案: 任意で「もういちど いれてね」の再入力ステップ（スキップ可でも可）。

### U9【低】コールドスタートでAPI往復が体感数秒
- 登録完了・リタイヤ保存などで Functions のコールドスタート待ちが数秒発生。
- 対応案: 楽観的UI（先に画面遷移し、同期は裏で）。登録は既にローカル先行だが、遷移前に
  サーバ待ちしている箇所を見直す。

### U10【低】登録に進捗表示（1/4 等）が無い
- あと何ステップか分からない。ステッパー（●●○○）を追加すると安心感が出る。

### U11【低】色選択の色名が2行に折り返す
- 「イエロー」「パープル」がカード内で2行になりレイアウトが乱れる。幅/フォント調整。

## 良かった点（維持したい）

- 演算ごとの色分けレベルカード。
- ○式（日本式 正解=○）の正答カウント表示。
- 正答時の即時自動判定＋緑フィードバック＋自動送り（テンポが良い）。
- 結果画面の祝福（🎉パーフェクト・新ベスト・あたらしいバッジ×N・もういちど みる）。
- 部分保存（リタイヤでも「ここまでの N問は きろくに のこるよ」）。
- PIN確定ボタン（自動送信廃止）と検証中ローディング。

## 変更計画（優先度順）

### フェーズ1: 子供向けの根幹UX（最優先）
1. **B2 誤答フィードバック**:
   入力桁数がその問題の答えの桁数に達したら自動判定する（正解=緑送り／不正解=赤表示）。
   不正解時は「もういちど」を促し、入力をクリアして再挑戦（スキップで次へは維持）。
   - 桁数判定は `answer` の桁数で行う（1桁問題は1桁入力で判定、2桁問題は2桁で判定）。
   - 対象: `app/sansu-100/play/page.tsx`（`handleInputChange`/`judge`）, `NumberKeypad`。
2. **B1 リタイヤ重複防止**:
   `useRef` による同期ガード（`finalizingRef.current` を onClick 冒頭で即 true）に変更し、
   2連打でも `finishSession()` が1回だけ走るようにする。完走側にも同じ ref を適用。

### フェーズ2: 直近変更の後始末・整合性
3. **U2 デイリーバッジの整理**: `badge-catalog.ts` からデイリー系を除去 or 再定義。
   既存ユーザーの獲得済みデータとの整合（表示のみ除外が安全）。
4. **U3 リタイヤの記録扱い見直し**: `isRetired` を `totalSessions`/バッジ母数から除外、
   もしくは「N問以上で1回」の閾値。履歴一覧には引き続き「途中」として残す。
5. **B3 debug封じ**: 本番では `?debug=1` を無効化（`NODE_ENV==='production'` のとき
   クエリを無視）。

### フェーズ3: 仕上げ・モチベーション
6. **U1 テーマ色の反映**（または登録ステップ削除）。
7. **U5 レベルカードに自己ベスト/クリア表示**、**U4 レベル並びの整理**。
8. **U6 グラフ空状態**、**U10 登録ステッパー**、**U7 登録ラベル**、**U11 色名折返し**。
9. **U9 体感速度**（楽観的UI）、**U8 PIN再入力確認**（任意）。

### 進め方
- フェーズ1を1PR（最優先・要テスト）、フェーズ2を1PR、フェーズ3は小粒なので
  まとめて1〜2PR。各PRはローカル＋Playwright（モバイル幅）で検証し、squash merge。
- マージ後は `?debug=1` 等を使った本番スモークと、管理CLIでのテストユーザー後始末を実施。
