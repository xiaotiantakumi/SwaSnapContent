# LOOP_PROMPT — 100マス計算 ゲーミフィケーション 自走ループ

あなたは 100マス計算アプリに「コイン経済 ＋ アバターショップ ＋ 複数アーケードゲーム」を、
**繰り返すほど良くなるように**段階実装する自走エージェントです。
**1周で進めるのは1タスクだけ**。状態は会話履歴ではなく、このディレクトリ
（`docs/sansu-100-gamification/`）のファイルが正です。`/compact` されてもここから復帰してください。

起動例（自走・インターバルなし）:
```
/loop docs/sansu-100-gamification/LOOP_PROMPT.md を実行
```

---

## 毎周の手順（イテレーション・プロトコル）

### 1. 復帰（現在地の把握）
- `BACKLOG.md`（TODOの正）、`JOURNAL.md` の末尾20〜40行、`PLAYBOOK.md`（罠と対策）を読む。
- 仕様が要るときだけ `PLAN.md` / `GAMES.md` を参照する（毎回全部は読まない）。
- 作業ブランチが `feat/sansu-100-gamification` であることを `git branch --show-current` で確認。違えば checkout。

### 2. 選択（1タスク）
- `BACKLOG.md` の未完了 `[ ]` のうち**最上位を1つだけ**選ぶ。依存（基盤→コイン→ショップ→ゲーム基盤→ゲーム本体）を尊重。
- スコープを広げない。途中で気づいた別作業は**実装せず** `BACKLOG.md` に新タスクとして積む。

### 3. 実装
- 選んだタスクのみ実装する。設計は `PLAN.md` / `GAMES.md` に従う。
- 型は `app/sansu-100/lib/types.ts` と `api/src/shared/sansuTypes.ts` の**2か所**を必ず同期。
- 追加フィールドは全て optional ＋ `?? デフォルト` で**後方互換**を保つ（既存ユーザーを壊さない）。
- コイン計算は `app/sansu-100/lib/coins.ts`（クライアント）と `api/src/shared/coins.ts`（サーバー）を
  **同一ロジックで複製**し、サーバーを確定値とする。複製ドリフトはテストで検出する。

### 4. 検証（ここが品質の要。ゲームは必ず複数回）
- 共通: `npm run lint && npm run build`。lib/ロジックを触ったら `npm test`。API を触ったら `cd api && npm run build`。
- **ゲーム/UI タスクは実ブラウザ確認が必須**。`npm run dev`（または `npm run swa:all`）を起動し、
  webapp-testing スキル か Claude_Preview MCP で対象画面を開き、**同じゲームを最低3回プレイ**して確認:
  - [ ] コンソールにエラーが出ない / クラッシュしない
  - [ ] 操作が効く（キーボード＋オンスクリーン両方）、ゲームオーバー→再開が成立する
  - [ ] 参加費コインが正しく引かれる。**報酬でコインが増えない**（増えていたらバグ）
  - [ ] 体感が軽い（カクつき・操作遅延・メモリ増の兆候がない）
- 失敗したら直して**再検証**。`PLAYBOOK.md` に既知の罠があれば先に潰す。
- **全部緑になるまで step 8（コミット）に進まない。**

#### ブラウザ検証の手順（実証済みレシピ／詳細は PLAYBOOK）
- フルスタックは `npm run sansu:dev`（azurite+API+next、:4280）をバックグラウンド起動し、`curl :4280/sansu-100` が
  200 になるまで待つ。終了は `npm run sansu:stop`。
- **サーバーロジック（コイン/購入/参加費）は curl で直接 API を叩いて検証**するのが最も確実
  （例: `/api/sansu/sessions` に POST→応答 `user.coins` を確認、同一 id 再送で二重加算しないこと）。
- **UI は Playwright MCP で確認**。ただし MCP ブラウザは Docker 内なので URL は `host.docker.internal:4280`。
  この origin は**非セキュアで Web Crypto が使えず、ユーザー作成/PIN がブラウザ経由では失敗する**。
  → ユーザーは `curl` で作るか、`localStorage` に直接注入する（**id 系キーは `JSON.stringify` して入れること**。
  `sansu-100:current-user` は `JSON.stringify('id')`、`sansu-100:users` は配列JSON、結果画面は
  `sessionStorage['sansu-100:last-result']` を注入）。注入後リロードして snapshot/screenshot で確認。

### 5. 記録（JOURNAL）
- `JOURNAL.md` に追記（テンプ레は同ファイル末尾参照）: 周番号・タスクID・やったこと・検証結果要約・残課題。
- 再発しうる学び（ビルドの罠、API の癖、検証の手順）は `PLAYBOOK.md` に1項目として足す。

### 6. TODO ブラッシュアップ（BACKLOG）
- 完了タスクを `[x]` にする。検証で見つかった新タスクを追加。
- 大きすぎたタスクは分割、不要になったものは削除。**進捗の正は常に BACKLOG**。

### 7. 自己改善（「繰り返すほど良くなる」の核）
- この周で**摩擦・手戻り・曖昧な指示・見落とし**があったか内省する。あれば**この LOOP_PROMPT を編集**して
  次周が改善するようにする（検証チェックの追記、罠の明文化、手順の具体化など）。
- 改善したら先頭の `loop-version` を +1 し、`JOURNAL.md` に「vN→vN+1: 何をなぜ変えたか」を残す。
- **やってよい改善**: 次周の品質/速度を上げる手順・チェック・罠の明文化。
- **やってはいけない**: ゴールや受入基準を勝手に緩めること。緩める必要を感じたら実装を止めてユーザーに相談。

### 8. コミット
- ビルド/テスト/動作確認が全て緑なら、`feat(sansu-100): <タスク要約>` で **1タスク=1コミット**。
- コミットメッセージ末尾に必ず:
  ```
  Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
  ```
- push はユーザーから指示があったときだけ。

### 9. 継続判定 ＆ 報告
- `BACKLOG.md` に未完了が残れば次周へ。全て `[x]` なら「**全タスク完了**」と報告してループ終了。
- 各周の最後に1行で報告: **「やったこと / 残タスク数 / 次にやること」**。

## 厳守事項
- **コミット前に必ずビルド・テスト**（CLAUDE.md 方針）。緑でないものはコミットしない。
- **1周1タスク**。迷ったら BACKLOG の最上位。
- ゲームは「**コインを使って遊ぶ**」もの。**報酬でコインを増やさない**（経済が壊れる）。報酬は限定バッジ/称号のみ。
- 子ども向け配慮: ガチャ排除・指名買い・没収やペナルティなし・🪙等アイコンで文字が読めなくても分かるUI。
- サーバー厳密同期: 残高/所持品はサーバー権威。クライアント申告を信じない。書込API応答の `user` で `saveUser` 上書き。

## 1ゲーム=数周で仕上げる（ゲーム本体タスクの型）
各アーケードゲームは独立ループ単位。1ゲームを A〜D の周に分けて仕上げ、不具合を持ち越さない:
- **周A**: 純粋ロジック `lib/games/<id>.ts`（React非依存）＋ユニットテスト
- **周B**: `games/<Id>Game.tsx`（Canvas2D + requestAnimationFrame 固定タイムステップ + 操作）
- **周C**: コイン消費（参加費/コンティニュー）＋報酬バッジ連携 ＋ **ブラウザ3回プレイ動作確認**
- **周D**: 仕上げ（効果音・難易度・モバイル十字・軽量化）＋再確認
新ゲーム追加は「`GAMES.md` に1エントリ書く → `BACKLOG.md` に周A〜Dの4タスクを積む」の定型で行う。
